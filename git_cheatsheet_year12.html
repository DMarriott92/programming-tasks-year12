How Git Works

Git is a version control system — it tracks every change you make to your code over time. Think of it like a detailed save history: you can always go back to any previous version, see exactly what changed, and why.

For this course, Git also lets your teacher review your work directly from your fork on GitHub without you emailing files around.

// The Three Locations

Understanding where your code lives at any moment is the key to understanding Git:

YOUR LAPTOP
Working Directory
Files you're actively editing in VS Code
→
STAGED
Staging Area
Changes marked ready to save
→
SAVED LOCALLY
Local Repository
Committed snapshots on your machine
→
GITHUB
Remote Repository
Your fork — visible to your teacher
📋 The Key Insight
Committing saves a snapshot to your local machine only. Pushing sends those commits to GitHub. Until you push, your teacher cannot see your work — always push before the deadline.

// Two Ways to Use Git

Every action in this guide can be done two ways. Use whichever feels more comfortable — they do exactly the same thing:

VS Code — click through the Source Control panel (the branch icon in the sidebar)
Git Bash / Terminal — type commands directly (open with Ctrl+` in VS Code)

First-Time Setup

You only need to do these steps once per laptop. They tell Git who you are so your commits are labelled correctly.

Set your identity
Attaches your name and email to every commit you make
once per machine
VS VS Code
  • Open the terminal with Ctrl+`
  • Type the bash commands on the right — there is no VS Code UI for this step
>_ Git Bash / Terminal
  • git config --global user.name "Your Name"
  • git config --global user.email "you@school.ac.uk"
  • Use your real name and school email address
Fork the repo
Creates your own personal copy of the teacher's repo under your GitHub account
once per course
VS VS Code
  • Go to the teacher's repo on GitHub in your browser
  • Click the Fork button at the top right
  • Select your own GitHub account as the destination
>_ Git Bash / Terminal
  • Forking is done on the GitHub website — there is no terminal command for this
  • After forking, use git clone (below) to get it onto your laptop
git clone
Downloads your fork from GitHub onto your laptop so you can work on it
once per machine
VS VS Code
  • Press Ctrl+Shift+P to open the command palette
  • Type Git: Clone and press Enter
  • Paste the URL of your fork (not the teacher's repo)
  • Choose a folder on your laptop to save it in
  • Click Open when VS Code asks if you want to open the cloned repo
>_ Git Bash / Terminal
  • Navigate to where you want the folder: cd Documents
  • git clone https://github.com/YOUR-USERNAME/programming-tasks-year12.git
  • Replace YOUR-USERNAME with your GitHub username
  • cd programming-tasks-year12 to enter the folder
⚠ Clone your fork, not the teacher's repo
The clone URL must come from your fork on GitHub. If you clone the teacher's repo directly, you won't be able to push your work back to GitHub.

The Daily Workflow

Every time you sit down to work on your tasks, the cycle is the same: pull any updates from GitHub, write your code, add your changes, commit them with a message, then push back to GitHub.

START
git pull
Get latest changes
→
WORK
edit files
Write your code
→
STAGE
git add
Mark changes ready
→
SAVE
git commit
Snapshot locally
→
SHARE
git push
Send to GitHub
git pull
Downloads any new commits from GitHub to your laptop — always do this before starting work
start of every session
VS VS Code
  • Click the ⟳ sync icon in the bottom-left status bar
  • Or open Source Control (Ctrl+Shift+G), click the ... menu, then Pull
>_ Git Bash / Terminal
  • git pull
git add
Stages your changes — marks which files should be included in the next commit
after editing files
VS VS Code
  • Open Source Control with Ctrl+Shift+G
  • Under Changes, click the + next to individual files to stage them
  • Or click + next to Changes to stage everything at once
  • Staged files move to the Staged Changes section
>_ Git Bash / Terminal
  • Stage a specific file: git add average_calculator.py
  • Stage everything changed: git add .
  • The . means "all changed files in this folder and below"
git commit
Saves a permanent snapshot of your staged changes with a message describing what you did
when a task is working
VS VS Code
  • Open Source Control with Ctrl+Shift+G
  • Type your commit message in the box at the top (e.g. Week 01 task 1 complete)
  • Press Ctrl+Enter or click the ✓ Commit button
>_ Git Bash / Terminal
  • git commit -m "Week 01 task 1 complete"
  • The -m flag lets you write the message inline
  • Without -m Git opens a text editor — just use -m

// Writing Good Commit Messages

❌ Bad✓ Good
stuffAdd average_calculator function
fixedFix off-by-one error in linear search
wk1Week 01 all core tests passing
aaaaaaAdd input validation to grade calculator
💡 Rule of Thumb
A good commit message completes the sentence: "If applied, this commit will..." — so "Add input validation" reads naturally, but "added stuff" does not.
git push
Sends your local commits up to GitHub — this is what makes your work visible to your teacher
end of every session
VS VS Code
  • Click the ⟳ sync icon in the bottom-left status bar
  • Or open Source Control, click ..., then Push
  • VS Code may ask you to sign in to GitHub the first time
>_ Git Bash / Terminal
  • git push
  • First time on a new machine you may need: git push -u origin main
🚨 Committed ≠ Pushed
Committing saves your work locally on your laptop only. If your laptop is lost, your work is gone. Push at the end of every session — it takes two seconds and means your work is safe on GitHub and visible to your teacher.

Checking Your Work

These commands help you understand what state your repo is in at any moment — what's changed, what's staged, and what's been committed.

git status
Shows which files have changed, which are staged, and which are untracked
whenever you're unsure
VS VS Code
  • Open Source Control with Ctrl+Shift+G
  • Changed files appear under Changes
  • Staged files appear under Staged Changes
  • The bottom status bar shows the current branch and sync state
>_ Git Bash / Terminal
  • git status
  • Red filenames = changed but not staged
  • Green filenames = staged and ready to commit
git log
Shows the history of all commits — who made them, when, and what the message was
to review your history
VS VS Code
  • Install the GitLens extension (already in your recommended extensions)
  • Click the GitLens icon in the sidebar to see a visual commit history
  • Or hover over any line of code — GitLens shows when it was last changed and by who
>_ Git Bash / Terminal
  • git log — full history (press q to exit)
  • git log --oneline — compact one-line-per-commit view
  • git log --oneline -5 — show only the last 5 commits

// What git log --oneline looks like

terminal output 
a3f92c1 Week 02 all core tests passing
7b1e4d8 Fix edge case in password strength checker
2c9a017 Add shopping list edit function
f84b3e2 Week 01 complete — all tests green
1a2b3c4 Initial commit
git diff
Shows exactly what has changed in your files since the last commit — line by line
to see what you changed
VS VS Code
  • Open Source Control with Ctrl+Shift+G
  • Click on any file listed under Changes
  • VS Code opens a side-by-side diff view — red = removed, green = added
>_ Git Bash / Terminal
  • git diff — see all unstaged changes
  • git diff average_calculator.py — diff a specific file
  • Lines starting with - were removed, + were added

Fixing Mistakes

Everyone makes mistakes. Git's job is to make them recoverable. Here are the situations you're most likely to encounter and how to fix them.

Unstage a file
You staged a file by mistake and want to remove it from the next commit without losing your changes
staged wrong file
VS VS Code
  • Open Source Control with Ctrl+Shift+G
  • Under Staged Changes, click the − next to the file
  • It moves back to Changes — your edits are still there
>_ Git Bash / Terminal
  • git restore --staged average_calculator.py
  • Your file changes are kept — only the staging is undone
Discard changes to a file
You've made a mess of a file and want to go back to the last committed version
file is broken beyond repair
VS VS Code
  • Open Source Control with Ctrl+Shift+G
  • Right-click the file under Changes
  • Select Discard Changes
>_ Git Bash / Terminal
  • git restore average_calculator.py
  • ⚠ This permanently discards your unsaved changes to that file
🚨 This Cannot Be Undone
Discarding changes removes them permanently — they were never committed so Git has nothing to restore from. Only do this if you are sure you want to throw away your edits to that file.
Fix the last commit message
You just committed but the message has a typo or is unclear
bad commit message
VS VS Code
  • Open Source Control, click ...
  • Select Commit → Undo Last Commit
  • Your changes come back as staged — re-commit with the correct message
>_ Git Bash / Terminal
  • git commit --amend -m "Corrected commit message"
  • Only use this if you have NOT yet pushed — amending a pushed commit causes problems
Pull rejected — local changes conflict
You tried to pull but Git says your local changes conflict with what's on GitHub
pull blocked by local edits
VS VS Code
  • Commit your local changes first (Ctrl+Shift+G → stage → commit)
  • Then pull again — VS Code will highlight any conflicts in the file
  • Click Accept Current or Accept Incoming for each conflict
  • Stage and commit the resolved file
>_ Git Bash / Terminal
  • Commit first: git add . && git commit -m "WIP"
  • Then: git pull
  • If conflicts appear, open the file — look for <<<<<<< markers
  • Edit the file to resolve, then: git add . && git commit -m "Resolve conflict"
💡 Best Prevention
The easiest way to avoid conflicts is to always pull before you start working and push when you finish. Conflicts mostly happen when you forget to pull and someone else (or you from another machine) has pushed in the meantime.
