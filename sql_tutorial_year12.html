What is SQL and Why Do We Use It?

SQL stands for Structured Query Language. It is the standard language for communicating with relational databases — systems that store data in organised tables, similar to a spreadsheet, but far more powerful.

Rather than storing data in a Python list or dictionary that disappears when your program ends, a database persists data permanently and lets you search, filter, and update it efficiently even when there are millions of records.

// Real-World Examples

ApplicationWhat SQL is Doing
School MIS systemStoring student records, timetables, attendance and grades — all linked together
Online shopManaging products, orders, customers and stock levels across related tables
Social mediaEvery post, like, follow and message stored and retrieved using SQL queries
Your Python programsWeeks 8, 9, 17, 18 — storing and retrieving data that outlives a single run of the program

// SQLite — SQL Without a Server

Most databases (MySQL, PostgreSQL) require a separate server process running in the background. SQLite is different — it stores the entire database in a single .db file on your computer, and Python has built-in support for it with no installation required.

💡 Why SQLite for A-Level?
SQLite is used in millions of real applications including iOS, Android, and Firefox. It is not a toy — it is simply a version of SQL that doesn't need a server, making it perfect for learning and for programs that run on a single machine.

What is Normalisation?

Normalisation is the process of organising a database to reduce redundancy (repeated data) and improve data integrity (keeping data accurate and consistent). The AQA spec covers up to Third Normal Form (3NF).

The best way to understand it is to see the problem it solves. Imagine storing student and subject data in a single flat table:

// The Problem — Unnormalised Data (UNF)

StudentIDStudentNameSubject1Teacher1Subject2Teacher2
1AliceMathsMr SmithPhysicsMs Jones
2BobMathsMr SmithChemistryDr Lee
3CarolPhysicsMs Jones

Problems: Mr Smith appears multiple times. If he changes his name, you must update every row. What if a student takes 5 subjects? You'd need to add more columns. This is fragile and wasteful.

// First Normal Form (1NF)

Each cell holds a single value, and each row is unique. We fix repeating groups by giving each student-subject pair its own row:

StudentIDStudentNameSubjectTeacher
1AliceMathsMr Smith
1AlicePhysicsMs Jones
2BobMathsMr Smith
2BobChemistryDr Lee

Better — but Alice's name is still repeated. And if Mr Smith's name changes, we still need to update multiple rows.

// Second & Third Normal Form — The Solution

Split the data into separate tables so each piece of information is stored exactly once, linked by keys:

students
student_idPK · INTEGER
nameTEXT
year_groupINTEGER
→
enrolments
enrolment_idPK · INTEGER
student_idFK → students
subject_idFK → subjects
←
subjects
subject_idPK · INTEGER
subject_nameTEXT
teacher_nameTEXT

Now Alice's name is stored once in students. Mr Smith's name is stored once in subjects. The enrolments table just records which student takes which subject using their IDs. Update Mr Smith in one place — it's instantly correct everywhere.

// Key Vocabulary

TermMeaning
Primary Key (PK)A unique identifier for each row — no two rows can share it. Usually an auto-incrementing integer.
Foreign Key (FK)A field in one table that references the primary key of another table — this is how tables link.
Referential integrityThe rule that a foreign key must always point to a row that actually exists.
NormalisationThe process of restructuring tables to remove redundancy and dependency issues.

Creating Tables in Python with sqlite3

Python's built-in sqlite3 module lets you create and interact with SQLite databases. No installation needed — it's part of the standard library.

// The sqlite3 Workflow

→

Run your SQL command
→

commit()
Save changes permanently
→
05

Close the connection
⚠ The Most Common Mistake
Forgetting to call commit() after INSERT, UPDATE or DELETE means your changes are never actually saved to the file. The program won't error — the data just silently disappears when you close the connection.

// Creating Your First Table

create_table.py 
import sqlite3

# Step 1: Connect to (or create) the database file
conn = sqlite3.connect('school.db')

# Step 2: Create a cursor
cursor = conn.cursor()

# Step 3: Execute a CREATE TABLE statement
cursor.execute('''
    CREATE TABLE IF NOT EXISTS students (
        student_id  INTEGER PRIMARY KEY AUTOINCREMENT,
        name        TEXT    NOT NULL,
        year_group  INTEGER NOT NULL
    )
''')

# Step 4: Commit and close
conn.commit()
conn.close()

print("Database and table created successfully.")

// Key Constraints

💡 Best Practice
Always use CREATE TABLE IF NOT EXISTS. Without it, running your script a second time will throw an error because the table already exists.

Writing Queries in Python

A query is a request sent to the database. The most important query is SELECT — it retrieves rows from a table based on conditions you define.

// Inserting Data First

Before you can query anything, you need data in your table. Use INSERT INTO:

insert_data.py 
import sqlite3

conn = sqlite3.connect('school.db')
cursor = conn.cursor()

# Insert a single student — use ? placeholders, NEVER string concatenation
cursor.execute(
    "INSERT INTO students (name, year_group) VALUES (?, ?)",
    ("Alice", 12)
)

# Insert multiple students at once
students = [
    ("Bob",   12),
    ("Carol", 13),
    ("David", 12),
]
cursor.executemany(
    "INSERT INTO students (name, year_group) VALUES (?, ?)",
    students
)

conn.commit()
conn.close()
🚨 SQL Injection — Never Do This
Building queries by joining strings with user input is a critical security vulnerability:

query = "SELECT * FROM students WHERE name = '" + user_input + "'"

A malicious user could type ' OR '1'='1 as their name and retrieve every row in the database. Always use ? placeholders instead.

// SELECT Queries

select_queries.py 
import sqlite3

conn   = sqlite3.connect('school.db')
cursor = conn.cursor()

# Get ALL rows from students
cursor.execute("SELECT * FROM students")

# Get only name and year_group columns
cursor.execute("SELECT name, year_group FROM students")

# Filter with WHERE
cursor.execute(
    "SELECT * FROM students WHERE year_group = ?",
    (12,)   # Note the trailing comma — this must be a tuple
)

# Sort results
cursor.execute("SELECT * FROM students ORDER BY name ASC")

# Limit results
cursor.execute("SELECT * FROM students LIMIT 5")

conn.close()

Using Query Results in Your Program

Running a SELECT query doesn't automatically show you the results — you need to fetch them. There are three ways to do this depending on how many rows you expect.

// The Three Fetch Methods

fetch_results.py 
import sqlite3

conn   = sqlite3.connect('school.db')
cursor = conn.cursor()

# ── fetchone() — get a single row ──────────────────────
cursor.execute("SELECT * FROM students WHERE student_id = ?", (1,))
student = cursor.fetchone()
# Returns a tuple: (1, 'Alice', 12)
# Returns None if no row is found

if student:
    print(f"Found: {student[1]}")   # student[1] is the name column
else:
    print("Student not found.")

# ── fetchall() — get all matching rows ─────────────────
cursor.execute("SELECT * FROM students WHERE year_group = ?", (12,))
all_year12 = cursor.fetchall()
# Returns a list of tuples: [(1,'Alice',12), (2,'Bob',12), ...]

for row in all_year12:
    print(f"ID: {row[0]}  Name: {row[1]}  Year: {row[2]}")

# ── fetchmany(n) — get the next n rows ─────────────────
cursor.execute("SELECT * FROM students")
first_three = cursor.fetchmany(3)

conn.close()

// Making Results Readable with Row Factories

Accessing results by index like row[1] works but is hard to read. Using row_factory lets you access columns by name instead:

row_factory.py 
import sqlite3

conn = sqlite3.connect('school.db')
conn.row_factory = sqlite3.Row    # ← add this line after connect()
cursor = conn.cursor()

cursor.execute("SELECT * FROM students")
students = cursor.fetchall()

for student in students:
    # Now access by column name instead of index
    print(f"{student['name']} is in Year {student['year_group']}")

conn.close()
💡 Which Method to Use?
Use fetchone() when looking up a specific record (e.g. login by username). Use fetchall() when you need to display or process a whole list. Use row_factory in any program you plan to show to someone else — it makes the code much easier to read.

Editing Data from Python

The remaining CRUD operations — Update and Delete — let you modify data that already exists in the database. Both follow the same pattern as INSERT: use ? placeholders, then commit().

// UPDATE — Modifying Existing Rows

update_data.py 
import sqlite3

conn   = sqlite3.connect('school.db')
cursor = conn.cursor()

# Update a single student's year group
cursor.execute(
    "UPDATE students SET year_group = ? WHERE student_id = ?",
    (13, 1)   # Set year_group to 13 where student_id is 1
)

# Update multiple columns at once
cursor.execute(
    "UPDATE students SET name = ?, year_group = ? WHERE student_id = ?",
    ("Alice Smith", 13, 1)
)

# Check how many rows were actually changed
print(f"Rows updated: {cursor.rowcount}")

conn.commit()
conn.close()
🚨 Always Use WHERE with UPDATE and DELETE
An UPDATE or DELETE without a WHERE clause affects every single row in the table. This is one of the most common and most costly mistakes in database programming. Always specify which rows you mean to change.

// DELETE — Removing Rows

delete_data.py 
import sqlite3

conn   = sqlite3.connect('school.db')
cursor = conn.cursor()

# Delete a specific student by ID
cursor.execute(
    "DELETE FROM students WHERE student_id = ?",
    (3,)
)

# Confirm the deletion
if cursor.rowcount == 1:
    print("Student deleted.")
else:
    print("No student found with that ID.")

conn.commit()
conn.close()

// A Complete CRUD Function Set

In your Week 09 task you'll build a modular system. Here's what a clean, modular approach looks like:

crud_module.py — modular pattern 
import sqlite3

DB = 'school.db'

def add_student(name, year_group):
    conn = sqlite3.connect(DB)
    conn.execute("INSERT INTO students (name, year_group) VALUES (?,?)", (name, year_group))
    conn.commit()
    conn.close()

def get_student(student_id):
    conn = sqlite3.connect(DB)
    conn.row_factory = sqlite3.Row
    row = conn.execute("SELECT * FROM students WHERE student_id=?", (student_id,)).fetchone()
    conn.close()
    return row

def update_year(student_id, new_year):
    conn = sqlite3.connect(DB)
    conn.execute("UPDATE students SET year_group=? WHERE student_id=?", (new_year, student_id))
    conn.commit()
    conn.close()

def delete_student(student_id):
    conn = sqlite3.connect(DB)
    conn.execute("DELETE FROM students WHERE student_id=?", (student_id,))
    conn.commit()
    conn.close()

Cross-Table Communication with JOIN

The real power of a relational database comes from linking tables together. A JOIN combines rows from two or more tables based on a matching key — letting you ask questions that span multiple tables in a single query.

Using the three-table structure from the Normalisation section:

students
student_idPK
nameTEXT
year_groupINTEGER
→
enrolments
enrolment_idPK
student_idFK
subject_idFK
←
subjects
subject_idPK
subject_nameTEXT
teacher_nameTEXT

// Creating the Tables in Python

create_relational.py 
import sqlite3

conn   = sqlite3.connect('school.db')
cursor = conn.cursor()

cursor.execute('''CREATE TABLE IF NOT EXISTS subjects (
    subject_id   INTEGER PRIMARY KEY AUTOINCREMENT,
    subject_name TEXT NOT NULL,
    teacher_name TEXT NOT NULL
)''')

cursor.execute('''CREATE TABLE IF NOT EXISTS enrolments (
    enrolment_id INTEGER PRIMARY KEY AUTOINCREMENT,
    student_id   INTEGER NOT NULL,
    subject_id   INTEGER NOT NULL,
    FOREIGN KEY (student_id) REFERENCES students(student_id),
    FOREIGN KEY (subject_id) REFERENCES subjects(subject_id)
)''')

conn.commit()
conn.close()

// INNER JOIN — Combining Tables

An INNER JOIN returns rows where there is a match in both tables:

join_queries.py 
import sqlite3

conn = sqlite3.connect('school.db')
conn.row_factory = sqlite3.Row
cursor = conn.cursor()

# Which subjects is each student enrolled in?
cursor.execute('''
    SELECT students.name, subjects.subject_name, subjects.teacher_name
    FROM enrolments
    INNER JOIN students ON enrolments.student_id = students.student_id
    INNER JOIN subjects ON enrolments.subject_id = subjects.subject_id
    ORDER BY students.name
''')

for row in cursor.fetchall():
    print(f"{row['name']} studies {row['subject_name']} with {row['teacher_name']}")

# How many students take each subject?
cursor.execute('''
    SELECT subjects.subject_name, COUNT(enrolments.student_id) AS student_count
    FROM subjects
    INNER JOIN enrolments ON subjects.subject_id = enrolments.subject_id
    GROUP BY subjects.subject_name
    ORDER BY student_count DESC
''')

for row in cursor.fetchall():
    print(f"{row['subject_name']}: {row['student_count']} students")

conn.close()

// Types of JOIN

JOIN TypeReturnsUse when
INNER JOINOnly rows with a match in both tablesMost common — you only want linked data
LEFT JOINAll rows from the left table, plus matches from the rightYou want all students even if not enrolled in anything
RIGHT JOINAll rows from the right table, plus matches from the leftLess common in SQLite

Aggregate functions like COUNT() and AVG() usually need a GROUP BY clause to tell SQL which column to group by. Without it, you get one result for the entire table rather than one per group.
