Writing Tests
Before You Write Code

A practical guide to pytest and test-driven thinking, using your Week 01 tasks as the example.

1

Why Write Tests Before Code?

Most beginners write code first and then manually check if it works — by running it, typing in some values, and eyeballing the output. That approach has some serious problems.

⚠ The Problem with Manual Testing
You only test the cases you think of in the moment. Edge cases — negative numbers, empty lists, unexpected inputs — get missed. Bugs hide there.

The professional approach is Test-Driven Development (TDD): write your tests first, then write the code that makes those tests pass. This forces you to think clearly about what your function should do before you worry about how.

The Three Benefits for A-Level

BenefitWhat it means for you
Clarity Writing a test forces you to define the exact inputs and expected outputs — that's essentially writing a specification before you code.
Confidence Once your tests pass, you know the function works — not just for the easy case, but for all the cases you defined.
Debugging speed When a test fails, pytest tells you exactly which function broke and what it returned instead of what you expected. No guessing.
💡 Good to Know
This technique is used in almost every professional software team. Learning it now puts you ahead of most programmers who only discovered it years into their careers.

How pytest Works

pytest is a Python testing library. You write functions that check whether your code behaves correctly, and pytest runs them all and reports the results.

// The Anatomy of a Test

example_test.py 
def test_celsius_to_fahrenheit_freezing_point():
    # Arrange: set up the input
    input_temp = 0

    # Act: call the function
    result = celsius_to_fahrenheit(input_temp)

    # Assert: check the output is what we expect
    assert result == pytest.approx(32.0)
📋 The Arrange–Act–Assert Pattern
Every test you write should follow this three-step pattern. Arrange your inputs, Act by calling the function, then Assert the output is correct. This keeps tests readable and easy to debug.

// Key pytest Tools You'll Use

ToolWhat it doesWhen to use it
assert x == y Checks two values are equal Most tests
pytest.approx(x) Checks a float is close enough Decimal results (e.g. 98.6°F)
isinstance(x, type) Checks the type of a value Confirming a function returns a float, list, etc.
pytest.raises(Error) Checks that an exception is raised Edge cases like empty lists

// Why pytest.approx() Matters

Floating point arithmetic in computers is not perfectly precise. For example:

Python interpreter 
>>> 0.1 + 0.2 == 0.3
False
>>> 0.1 + 0.2
0.30000000000000004

So for any test involving decimal division or temperature formulas, always use pytest.approx() instead of ==.

Filling in the TODOs

Every # TODO in the file is a gap you need to fill. Here's how to think through each one, using real examples from your test file.

// Example 1: A Basic assert

Before — incomplete 
    def test_average_of_two_numbers(self):
        # TODO: call calculate_average() with two numbers that give a whole number average
        assert calculate_average(  # TODO
        ) == # TODO

Think: what two numbers have a whole-number average? 4 and 8 average to 6. Fill it in:

After — complete ✓ 
    def test_average_of_two_numbers(self):
        assert calculate_average([4, 8]) == 6.0

// Example 2: Checking the Return Type

Before — incomplete 
    def test_average_returns_float_type(self):
        result = calculate_average(  # TODO
        )
        assert isinstance(result, # TODO
        )

You need any valid input, then check the type is float:

After — complete ✓ 
    def test_average_returns_float_type(self):
        result = calculate_average([1, 2, 3])
        assert isinstance(result, float)

// Example 3: Decimals with pytest.approx()

Before — incomplete 
    def test_average_decimal_result(self):
        assert calculate_average(  # TODO
        ) == pytest.approx(  # TODO  ← use pytest.approx() for decimals
        )

The average of 1 and 2 is 1.5 — a decimal. Use pytest.approx():

After — complete ✓ 
    def test_average_decimal_result(self):
        assert calculate_average([1, 2]) == pytest.approx(1.5)

// Example 4: Testing That an Error is Raised

Before — incomplete 
    def test_average_empty_list_raises_error(self):
        with pytest.raises(  # TODO: ValueError or another appropriate exception
        ):
            calculate_average([])

An empty list has no average. Your function should raise a ValueError for this — so the test checks that it does:

After — complete ✓ 
    def test_average_empty_list_raises_error(self):
        with pytest.raises(ValueError):
            calculate_average([])
📋 Note
Writing this test before you write calculate_average() is the point — it forces you to decide: should I raise a ValueError or return None? Making that decision consciously leads to better code.

Running Your Tests

Once you've filled in the TODOs and written your actual Python functions, open the VS Code terminal and run:

Terminal (PowerShell or bash) 
python -m pytest test_week01.py -v

The -v flag means "verbose" — it prints the name of every test so you can see exactly what passed and what failed.

// Reading the Output

collected 34 items

PASSED test_week01.py::TestAverageCalculator::test_average_of_three_numbers
PASSED test_week01.py::TestAverageCalculator::test_average_of_two_numbers
FAILED test_week01.py::TestAverageCalculator::test_average_with_negative_numbers
PASSED test_week01.py::TestMinMaxFinder::test_find_max_basic
...

FAILED test_week01.py::TestAverageCalculator::test_average_with_negative_numbers
─────────────────────────────────────────
AssertionError: assert -2.0 == 2.0
Where calculate_average([-6, 2]) returned -2.0

short test summary info
FAILED - 1 failed, 3 passed in 0.12s

This output immediately tells you: the test for negative numbers failed because the function returned -2.0 when you expected 2.0. You'd then go check your logic — and in this case, perhaps your test expectation was wrong, not the function!

💡 Red–Green–Refactor
The professional TDD loop is: write a test → watch it fail (red) → write the code → watch it pass (green) → tidy up → repeat. Red is not a bad thing — it means your test is working.
